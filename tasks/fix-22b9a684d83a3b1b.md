# fix-22b9a684d83a3b1b — Invite counter key has 7-day TTL — invite codes can recycle after expiry

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 SBOT

`nextCode()` in `src/storage/invite-store.ts:58` writes the counter key with `DEFAULT_TTL` (7 days). If the counter expires before any active invite codes referencing the recycled range are still alive, newly generated codes can collide with recently-expired codes that may still be floating in transit. A counter that must be monotonically increasing should not have a TTL — it causes the counter to reset. While both the counter and individual invites share the same 7-day TTL, Redis expiry is lazy (not exact), so a timing skew could let an old invite briefly coexist with a recycled counter.

Fix: store the counter without a TTL, or use a time-prefixed counter to avoid collision.
